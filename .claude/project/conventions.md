# Project Memory — Automation Conventions

How automation code is written in **this** repository. This is the code-generation
contract the automation phases follow. Portable *methodology* lives in Skills
(`.claude/skills/`); reusable *coverage* lives in Checklists (`.claude/checklists/`);
*stack facts* live in [stack.md](stack.md).

- Stack / registry / env / run matrix → [stack.md](stack.md)
- Full architecture narrative (human) → root `CLAUDE.md`

## Where automation code is written (per app)

Generate into the target app under `apps/<app>/`:

- `pom/*.page.ts` — UI Page Objects  ·  `som/*.api.ts` — API Service Objects
- `features/*.feature` — Gherkin (**UI only**)  ·  `steps/*.steps.ts` — cross-fixture orchestration
- `specs/e2e|api/*.spec.ts` — spec tests  ·  `specs/fixtures.ts` + `steps/fixtures.ts` — fixtures
- `apps/<app>/.env` (+ `.env.example`) — URLs/secrets  ·  `.features-gen/<app>/` — generated by `bddgen`, never edit

## Code-generation standards

- **Reuse before creating**: existing features, step decorators, POM/SOM, fixtures, hooks, helpers, selectors, and schemas. Never duplicate. Inspect only the relevant `apps/<app>/` files.
- **API is spec-only**: verify API behavior in `specs/api/*.spec.ts` (via SOM action methods). **Never** author an API `.feature` file — feature files are for UI behavior. An API SOM keeps its `@Given/@When/@Then` decorators only so the service object can back **UI** BDD steps (setup/teardown/regular steps) when an app also has a UI.
- **Dual-style classes**: every POM/SOM exposes a plain helper method **and** a thin `@Given/@When/@Then` decorator over it — no standalone single-fixture step files. Register each new fixture in **both** `specs/fixtures.ts` and `steps/fixtures.ts`. (For an API SOM the decorators are only consumed when the app's UI features use them.)
- **POM three-section layout** (`Assertions` / `Actions` / `BDD step decorators`) — reference [`apps/saucedemo/pom/login.page.ts`](../../apps/saucedemo/pom/login.page.ts).
- **Relative navigation** (`page.goto('/')`, `request.get('/path')`); base URL comes from `app.config.ts` / `apps/<app>/.env`; for an API base path (e.g. `/v2`) put the origin in `baseURL` and the prefix on endpoint paths.
- **Locators**: `getByRole` / `getByLabel` / `getByTestId`. Auto-retrying assertions; no hard waits.
- Every `@Then` uses the verb **"should"**.
- **Tags — UI Gherkin**: `@smoke` (must-pass gate) · `@regression` · `@ui`. `@triage` marks a failed-case reproduction — excluded from smoke/regression, run via `npm run test:triage` / `npm run test:bdd:triage`; never also tag it `@smoke`/`@regression`. `@manual` marks a scenario authored by `/manual-qa` whose steps are not yet implemented — it carries no run tags (so `bddgen` tag expressions never select it) until `/auto-qa` implements the steps and retags it.
- **Tags — API specs**: use Playwright's native tag option, `test('...', { tag: '@smoke' }, ...)` or `test.describe('...', { tag: '@smoke' }, ...)` — `@smoke` · `@regression` (regression = the union, `--grep "@smoke|@regression"`; see `test:api:smoke` / `test:api:regression` in [stack.md](stack.md)). Every real test carries one tag; a `@triage` API repro is a spec test tagged `@triage` (excluded from both, run via `test:triage`), never also `@smoke`/`@regression`.
- Keep Gherkin business-readable; steps thin (UI behavior in POM). API behavior lives in SOM methods called from specs.

## BDD file naming (UI only)

Files in `apps/<name>/features/` and `steps/` are named by **behavior/functionality**,
never by workflow letter. `steps/` holds only `fixtures.ts`, `hooks.ts`, and
cross-fixture orchestration step files (e.g. `checkout-setup.steps.ts`). Single-fixture
steps live as decorators on the POM class. (Feature files exist only for UI apps; API
apps have `som/` + `specs/api/` and no `features/`.)

## Feature file structure rules (UI only)

- **Single-scenario feature**: no `Background` — inline the setup steps.
- **Given vs When**: if a scenario has no `When` of its own, write its `Given` as `When` (unless the `Background` already contains a `When`).
- For multi-step setup that isn't itself the test, use a composite `@Given` in a `steps/*.steps.ts` orchestration file rather than a long `Background` of granular steps.

## Validate before done

Run `npx tsc --noEmit` and the relevant `npm run test:<app>` (plus `npm run
test:bdd:<app>` for UI apps; full matrix in [stack.md](stack.md)). Report the run
command and a summary of files created/modified.

## Review gate

The workflow is strictly sequential: `/auto-qa` writes code **only** from the reviewed
manual specification (`deliverables/<feature>/02-Manual-QA.md` + the Gherkin it
created/updated). Running a stage command implicitly accepts the previous stage's
deliverable; each command then owns its own interactive review (the `qa-review` skill).
